Loop Engineering：把“继续 prompt”变成可验证的反馈系统

它不是魔法新范式。冷静看，它是 agent loop、automation、外置状态、验证器和权限边界的组合。真正的问题不是“能不能自动跑”，而是“跑错时谁发现、谁付钱、谁负责”。

当前判断：适合低风险、反馈明确、可拆解、可回滚的工程任务；不适合拿来遮盖需求不清、验证薄弱和人工责任缺位。

一句话区别

Prompt Engineering：这一轮怎么问模型。

Loop Engineering：下一轮由谁触发、读什么状态、怎么验证、何时停止、失败后怎么处理。

“Loop changes the work, it does not delete you from it.” — Addy Osmani

几篇文章分别补了什么

最适合做工程拆解。它把 loop 的基础设施讲清楚，但结尾也明确提醒：loop 改变工作，不删除工程师。

最适合做分层判断。很多人只做了 agent loop，却以为自己在做 event-driven 或 hill-climbing loop。越往外层，复利越强，治理成本也越高。

最适合降温。它提醒 loop 叙事也服务 token 消费。供应商喜欢“自动跑更久”，工程团队需要预算、审计和人审。

最适合反证。planner → executor → critic → repair 看起来完整，但 critic 持续打回而 executor 无法改善时，loop 只会增加成本和延迟。

最适合看组织瓶颈。代码变便宜以后，spec、上下文、团队共识会变成新瓶颈。没有外置上下文，loop 只是更快地做错题。

第一版建议

  1. 从 CI 归因、文档链接修复、issue triage 这类低风险闭环开始。
  2. 停止条件必须是机器可检查的信号。
  3. 每轮写外置状态：尝试、证据、失败原因、下一步。
  4. maker 和 checker 分离，别让生成者自己宣布成功。
  5. 先产出建议或 PR 草稿，不自动 merge。

上线前追问

  1. 谁触发？读什么？能写哪里？
  2. 失败后怎么避免重复同错？
  3. token、时间、并发、重试有没有硬上限？
  4. 人在哪些点必须介入？能看到哪些证据？
  5. 写坏了怎么回滚？模型换了还能跑吗？
